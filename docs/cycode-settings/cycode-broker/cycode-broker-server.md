## Overview [Copied!](#overview "Copy link to this section")

The Cycode Broker server service does the following:

*   Saves tasks to be executed by the broker client
    
*   Receives the task results from the broker client
    
*   Stores logs and analytics on the broker clients
    

## Security features [Copied!](#security-features "Copy link to this section")

The Cycode Broker server implements multiple layers of security to protect customer data throughout the scan lifecycle:

*   **Encryption in transit** - All connections use TLS 1.2+ with optional mutual TLS (mTLS) for bidirectional certificate authentication.
    
*   **Encryption at rest** - Clone artifacts are encrypted using hybrid RSA-2048/AES-256 before upload and stored with SSE-KMS in S3. MongoDB sensitive fields use client-side field-level encryption (CSFLE).
    
*   **Ephemeral data** - Cloned repositories are automatically deleted from local disk after upload and from S3 after a configurable TTL (default 24 hours).
    
*   **Scoped credentials** - Temporary S3 credentials are generated per clone request. The credentials are scoped to the tenant’s S3 path and expire after 6 hours.
    
*   **BYOS & CMK** - Enterprises can use [Bring Your Own Storage (BYOS)](/integrations/scm-source-code-management-tools/bring-your-own-storage-byos) and Customer Managed Keys (CMK) to keep data and encryption keys in their own AWS account.
    

Note

For more information, go to [Cycode Broker security architecture](/cycode-settings/cycode-broker/cycode-broker-security-architecture) .

## Cycode Broker client-server connection [Copied!](#cycode-broker-client-server-connection "Copy link to this section")

The Cycode Broker client has to authenticate itself successfully to the Cycode Broker server, and then uses a long polling approach to receive tasks for execution. The process is illustrated in the diagram and described below.

1.  The Cycode Broker client opens a long-lived HTTPS connection to the Cycode Broker server.
    
2.  The Cycode Broker server receives the HTTP connection and begins polling for tasks.
    
3.  If there are no tasks to deliver, the connection remains open until it reaches the HTTPS timeout.
    
4.  The Cycode Broker client reaches the HTTPS timeout.
    
5.  The Cycode Broker client immediately opens another connection and waits for tasks.
    
6.  When the Cycode Broker server is populated with tasks, it sends them in the open HTTPS connection. This connection is then closed.
    
7.  The Cycode Broker client reviews the tasks, executes them, and sends the response to the Cycode Broker server.
    
8.  The Cycode Broker client opens another HTTPS connection to wait for additional tasks.
    

## Clone tasks [Copied!](#clone-tasks "Copy link to this section")

REST-based tasks are generally short and don’t contain a lot of data. These tasks are executed as soon as the Cycode Broker client receives them, and the response is sent via POST to the Cycode Broker server.

Clone tasks, on the other hand, can take much longer and can may also be extremely heavy for large repositories. Therefore, the clone mechanism in the Cycode Broker is different.

1.  The Cycode Broker client receives a clone task from the broker server. This clone task contains temporary S3 credentials generated by the Cycode Broker server for each clone request. The credentials are scoped to the tenant’s S3 path, expire after 6 hours, and are restricted to upload-only permissions (they can’t be used to read from the bucket).
    
2.  The Cycode Broker client stores the task.
    
3.  The Cycode Broker client utilizes a background worker that checks whether a clone task is waiting in the queue. This ensures that clone tasks don’t block the ongoing REST/GraphQL tasks that are being sent to the Cycode Broker client.
    
4.  If a clone task exists, the clone worker contacts the internal integration with the given credentials and clones the repository.
    
5.  After the repository is cloned, the worker zips and encrypts the contents (hybrid RSA-2048/AES-256), and uploads the zip file to a dedicated S3 bucket using the temporary S3 credentials. The S3 bucket uses SSE-KMS encryption at rest.
    
    Note
    
    If you have BYOS implemented, the worker uploads the zipped content to the defined S3 bucket instead. If you have CMK implemented, the specified KMS key is used to encrypt the contents. For more information about BYOS, go to [Bring Your Own Storage (BYOS)](/integrations/scm-source-code-management-tools/bring-your-own-storage-byos) .
    
6.  The clone worker removes the cloned repository and zip file from the local machine and continues to the next clone task (if one exists). Objects that were uploaded to the S3 bucket are tagged with a configurable TTL (default is 24 hours) and automatically deleted via S3 lifecycle rules. The default TTL provides the following advantages:
    
    *   Optimizes performance by caching clones to avoid re-cloning the same repository multiple times within a short timeframe.
    *   Ensures that data is ephemeral.
